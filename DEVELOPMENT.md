## 2. 詰まった点とその解決方法

### Next.js 16 の params は非同期

**問題：** Next.js 16では動的ルートの `params` が `Promise` 型になり、直接アクセスするとエラーになる。

**解決：**
```ts
// ❌ 旧来の書き方
export default function Page({ params }: { params: { id: string } }) {
  const { id } = params
}

// ✅ Next.js 16の書き方
export default async function Page({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params
}
```

---

### Supabase RLS でデータが取得できない

**問題：** RLSポリシーを設定後、認証済みユーザーでもデータが取得できなくなった。

**解決：** `SELECT` ポリシーに `authenticated` ロールへの許可を追加。開発中はメール確認を無効化して動作確認。

---

### iOS Safari の日付入力欄がはみ出す

**問題：** iPhoneでは `input[type="date"]` が `width: 100%` を無視してはみ出す。

**解決：** CSSに `-webkit-appearance: none` と `box-sizing: border-box` を追加。さらに日付フィールドのコンテナを縦並び固定（`flex-col`）に変更することで完全解決。

---

### ハイドレーションエラー

**問題：** ダークモードの `color-scheme` をJSでDOM操作した際にSSR/CSRの不一致が発生。

**解決：** DOM直接操作をやめ、`<style>` タグの動的注入に変更。ライトモード時はstyleタグを削除するだけにすることでサーバーとクライアントの一致を維持。

---

### Vercelデプロイ時のビルドエラー

**問題：** TypeScriptの型エラー（`Profile` 型の `full_name` vs `name`）でビルドが失敗。

**解決：** `src/types/index.ts` の型定義を確認し、`Sidebar.tsx` 内の `full_name` を `name` に統一。

---

### VitestがPlaywrightのテストファイルを拾う

**問題：** `tests/` フォルダのPlaywrightテストがVitestに検出され、CI環境でエラーになった。

**解決：** `vitest.config.ts` に `include: ['src/test/**/*.test.ts']` を追加し、Vitestの対象ファイルを明示的に指定。

---

### [id]/page.tsx に一覧ページの内容が上書きされた

**問題：** ファイルをダウンロードして差し替える作業中に、`reports/page.tsx` の内容を誤って `[id]/page.tsx` に配置してしまい、日報詳細画面が日報一覧を表示するようになった。

**解決：** Pythonスクリプトでターミナルから直接正しいコードをファイルに書き込んで解決。ファイル管理の重要性を学んだ。

---

### ログイン画面がアプリのダークモード設定に影響される

**問題：** アプリ内でダークモードに切り替えると `data-theme="dark"` が `html` 要素に設定され、ログイン画面にも影響してしまう。ログイン画面はシステム設定のみに従わせたかった。

**解決：** ログイン画面のマウント時に `useEffect` で `data-theme` 属性を一時的に除去し、アンマウント時に復元する方式を採用。`globals.css` に `html:not([data-theme])` セレクターでシステムのダークモードに追従するCSSを追記。

---

### E2EテストのCI実行でサーバーが起動していない

**問題：** GitHub Actions のCI環境でPlaywrightテストを実行すると `ERR_CONNECTION_REFUSED` が発生。ローカルでは `npm run dev` を別ターミナルで起動しているが、CI環境では起動されていなかった。

**解決：** `ci.yml` のE2Eジョブで `npm run build` 後に `npm run start &` でサーバーをバックグラウンド起動し、`wait-on` パッケージでサーバーの起動を待ってからテストを実行する方式を採用。

## 3. 技術選定の理由

| 技術 | 選定理由 |
|---|---|
| Next.js 16（App Router） | サーバーコンポーネントによるパフォーマンス最適化・SEO対応 |

## 4. 設計判断の理由

### App Router を採用

Pages Routerより新しいApp Routerを採用。サーバーコンポーネントで認証チェックをサーバー側で行い、セキュリティを強化。未ログイン時のリダイレクトもサーバー側で完結する。

### Supabase RLS による認可

APIルートを作らず、Supabase RLSでレコードレベルの権限制御を実装。自分の日報・コメントのみ編集・削除可能にした。バックエンドのコード量を削減しつつ、セキュリティを担保。

### リアルタイムバリデーション

`onBlur` イベントでバリデーションを実行。入力中はエラーを出さず、フォーカスを外したタイミングで表示することでUXを向上。文字数カウンターは上限に近づくとオレンジ→赤に変化。

### ダークモードの実装方針

`localStorage` に設定を保存し、`data-theme` 属性でCSSを切り替える方式を採用。Tailwindの `dark:` クラスではなくCSS変数で制御することでSSRとの競合を回避。

ログイン・登録画面はSidebarがないためアプリ内切り替えができない。ログイン画面マウント時に `data-theme` を除去し、`html:not([data-theme])` セレクターと `@media (prefers-color-scheme: dark)` の組み合わせでシステム設定のみに追従させる方式を採用。

### バリデーション関数の分離（Week7対応）

テストカバレッジを計測するため、バリデーション関数を `src/lib/validation.ts` に切り出した。テストファイルに関数を直接書く方式ではカバレッジが0%になることを経験し、ソースコードとテストを分離する重要性を学んだ。

### Git Flow 戦略

機能ごとにブランチを切り（`feature/xxx`）、`main` への直接コミットを避けた。PRを通じたコードレビューを意識した開発フローを実践。

## 5. テスト戦略（Week7・8）

### ユニットテスト（Vitest）

`src/lib/validation.ts` のバリデーション関数を対象に15件のテストを実装。カバレッジ100%を達成。

| テスト対象 | 件数 | 内容 |
|---|---|---|
| validateTitle | 5件 | 空文字・スペース・文字数超過・正常系 |
| validateContent | 4件 | 空文字・文字数超過・正常系 |
| validateEmail | 3件 | 空文字・形式不正・正常系 |
| validatePassword | 3件 | 空文字・文字数不足・正常系 |

### E2Eテスト（Playwright）

実際のブラウザでユーザーフローを検証する4シナリオを実装。

| シナリオ | 内容 |
|---|---|
| ログインできる | メール・パスワード入力→日報一覧へ遷移 |
| 日報を作成できる | フォーム入力→作成→一覧に表示 |
| 日報詳細を表示できる | 一覧のカードクリック→詳細画面へ遷移 |
| 未ログイン時リダイレクト | /reports アクセス→/login へリダイレクト |

### CI/CD（GitHub Actions）

`push` および `PR作成時` に以下が自動実行される。

- ユニットテスト（Vitest）+ カバレッジ計測
- ビルド・型チェック（`npm run build`）
- E2Eテスト（Playwright）※ `wait-on` でサーバー起動待機後に実行
