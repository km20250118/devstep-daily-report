## 2. 詰まった点とその解決方法

### Next.js 16 の params は非同期

**問題：** Next.js 16では動的ルートの params が Promise型になり、直接アクセスするとエラーになる。

**解決：** await params で非同期取得するよう修正。

### Supabase RLS でデータが取得できない

**問題：** RLSポリシーを設定後、認証済みユーザーでもデータが取得できなくなった。

**解決：** SELECT ポリシーに authenticated ロールへの許可を追加。

### iOS Safari の日付入力欄がはみ出す

**問題：** iPhoneでは input[type=date] が width:100% を無視してはみ出す。

**解決：** CSSに -webkit-appearance:none と box-sizing:border-box を追加。

### ハイドレーションエラー

**問題：** ダークモードの color-scheme をJSでDOM操作した際にSSR/CSRの不一致が発生。

**解決：** DOM直接操作をやめ、style タグの動的注入に変更。

### Vercelデプロイ時のビルドエラー

**問題：** TypeScriptの型エラー（Profile型の full_name vs name）でビルドが失敗。

**解決：** src/types/index.ts の型定義を確認し、使用箇所を統一。

## 3. 技術選定の理由

| 技術 | 選定理由 |
|---|---|
| Next.js 16（App Router） | サーバーコンポーネントによるパフォーマンス最適化・SEO対応 |
| TypeScript | 型安全性によるバグ防止・コードの可読性向上 |
| Supabase | PostgreSQL・認証・ストレージが一体化しており開発効率が高い |
| Tailwind CSS | ユーティリティクラスで素早くUIを構築できる |
| shadcn/ui | アクセシビリティ対応済みのコンポーネントをカスタマイズして使用 |
| Vercel | Next.jsとの親和性が高く、GitHubと連携した自動デプロイが容易 |
| Vitest | Viteベースで高速・Next.jsプロジェクトとの相性が良い |
| Playwright | ブラウザ実機でのE2Eテストが可能 |

## 4. 設計判断の理由

### App Router を採用

サーバーコンポーネントで認証チェックをサーバー側で行い、セキュリティを強化。

### Supabase RLS による認可

APIルートを作らず、Supabase RLSでレコードレベルの権限制御を実装。自分の日報・コメントのみ編集・削除可能にした。

### リアルタイムバリデーション

onBlur イベントでバリデーションを実行。入力中はエラーを出さず、フォーカスを外したタイミングで表示することでUXを向上。

### ダークモードの実装方針

localStorage に設定を保存し、data-theme 属性でCSSを切り替える方式を採用。SSRとの競合を回避。

### Git Flow 戦略

機能ごとにブランチを切り、main への直接コミットを避けた。PRを通じたコードレビューを意識した開発フローを実践。
